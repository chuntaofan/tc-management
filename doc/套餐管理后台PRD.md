## 4. 需求背景

### 4.1 背景

TeleAgent 商业化套餐需要一个后台管理能力，用于维护前台可售的个人套餐、企业团购和积分加油包。当前项目聚焦「计费管理 > 套餐管理」的 MVP，不覆盖完整下单、续订、退订、订单对账和权益履约，仅解决运营后台中套餐商品配置与版本发布的问题。

在套餐管理场景中，「上架时间」不是单纯的开关字段。只要存在待上架套餐，就必须明确它上线后替换哪个旧套餐，否则会出现同一套餐位下多个版本并列可售的问题。例如，运营新增一个新的个人基础版，如果没有替换对象，系统无法判断它是一个全新商品，还是当前个人基础版的下一版。

因此，本期需要把套餐管理从“单条套餐记录上下架”收敛为“同一售卖位下的版本发布”：运营通过新增或复制创建一个待上架版本，选择同类型的当前已上架套餐作为替换对象，到达上架时间后新版本生效，旧版本进入历史状态。

### 4.2 目标用户

目标用户为 TeleAgent 运营管理员，主要使用场景包括：

- 查看当前后台维护的套餐商品。
- 新增一个可售套餐或待上架套餐。
- 为已有套餐创建下一版配置。
- 管理套餐的已上架、待上架、已下架、已被替换状态。
- 避免直接修改已上线套餐，保证历史配置可追溯。

### 4.3 当前项目现状

当前项目已实现套餐管理页面，数据通过 localStorage mock 持久化，页面能力包括：

- 套餐列表展示。
- 关键词、套餐类型、状态筛选。
- 新增套餐。
- 仅待上架套餐可编辑。
- 已上架套餐可下架，待上架套餐可取消上架。
- 已下架/已被替换套餐不可重新上架，只能复制后生成新版本。
- 待上架套餐可选择同类型替换对象，到点后自动替换旧版本。

当前限制：

- 本期仅支持自有渠道，渠道字段固定为 `own_channel`。
- SKU 目前为前端静态配置，未接入业务支撑系统。
- 生命周期同步当前由前端查询列表时触发，后续接真实后端时建议迁移为服务端任务或服务端查询时计算。
- 当前 MVP 允许删除任意状态套餐；真实业务中建议后续限制已上架/已被替换套餐删除。

### 4.4 产品方案

本期通过「套餐类型 + 售卖位 `seriesKey` + 替换对象 `replacesPackageId`」实现简单版本发布模型：

- 同一售卖位代表一个稳定的前台销售位置，例如个人基础版、个人标准版、企业团购、加油包。
- 新增未来上架套餐时，必须选择同类型的当前已上架套餐作为替换对象。
- 待上架套餐到达上架时间后，系统将新套餐置为已上架，并将旧套餐置为已被替换。
- 已上架、已下架、已被替换套餐均不可编辑。
- 已下架套餐不允许重新上架；运营如需再次发布，只能复制生成新的待上架版本。

## 5. 预估收益

### 5.1 定性收益

- 降低运营误操作风险，避免直接编辑线上可售套餐。
- 明确套餐版本关系，解决“待上架套餐上线后替换谁”的业务歧义。
- 保留历史套餐配置，便于后续订单、权益、审计追溯。
- 为后续接入真实后端、订单系统和权益履约系统预留清晰的数据模型。

### 5.2 定量收益

当前 MVP 暂无线上统计数据。后续可通过以下指标评估：

- 套餐配置错误率。
- 套餐版本发布平均耗时。
- 套餐发布后回滚或修复次数。
- 运营创建新版本的操作完成时长。

## 7. 需求详情

### 7.1 业务流程

1. 运营进入 `TeleAgent > 计费管理 > 套餐管理`。
2. 系统初始化套餐数据，并同步待上架套餐生命周期。
3. 页面展示套餐列表，运营可按关键词、套餐类型、状态筛选。
4. 运营可点击「新增套餐」创建一条套餐记录。
5. 如果新增套餐的上架时间为未来时间，则该套餐进入待上架状态，必须选择同类型替换对象。
6. 待上架套餐在到达上架时间后自动变为已上架，被替换套餐变为已被替换。
7. 已上架套餐可下架；待上架套餐可取消上架；两者操作后状态均变为已下架。
8. 已下架和已被替换套餐不可编辑、不可重新上架，仅支持复制。
9. 复制会打开新增套餐弹窗，带入旧套餐配置，运营调整后提交为新记录。

### 7.2 页面结构

页面采用后台管理布局，整体从上到下包括：

- 面包屑：`TeleAgent > 计费管理 > 套餐管理`。
- 标题区：标题为「套餐管理」，右侧展示「新增套餐」按钮。
- 页面说明：说明该页面用于商业化套餐配置。
- 筛选区：关键词、套餐类型、状态、查询、重置。
- 表格区：展示套餐列表和状态化操作按钮。
- 弹窗：用于新增、编辑、复制套餐。

### 7.3 列表筛选

筛选区位于列表上方，支持运营按套餐名称、类型和状态快速定位套餐。

- 关键词：输入框。支持按套餐名称、SKU、权益说明进行模糊搜索。
- 套餐类型：下拉选择。可选择个人套餐、企业团购、加油包。
- 状态：下拉选择。可选择已上架、待上架、已下架、已被替换。
- 查询：点击后按当前筛选条件刷新列表。
- 重置：点击后清空全部筛选项，并重新展示全部套餐。

### 7.4 表格字段

表格用于展示套餐配置的核心信息，并根据状态展示可用操作。

- 套餐名称：展示套餐配置名称。
- SKU：仅展示中文 SKU 名称，例如“个人基础套餐”。底层仍保存对应 `skuCode`，用于唯一性校验和后续业务系统对接。
- 类型：展示个人套餐、企业团购或加油包。
- 套餐内容：展示权益说明文本，内容过长时省略。
- 有效期：个人套餐和加油包展示自然月、自然年或指定天数；企业团购本期展示 `-`。
- 刊例价：个人套餐和加油包展示刊例价；企业团购展示元/人/月。
- 实际售价：个人套餐和加油包展示实际售价；企业团购展示元/人/月。
- 折扣：根据实际售价和刊例价计算。实际售价为 0 时展示“免费”；企业团购展示 `-`。
- 渠道：当前固定展示自有渠道。
- 上架时间：格式为 `YYYY-MM-DD HH:mm`。
- 状态：展示已上架、待上架、已下架或已被替换。
- 操作：根据状态展示编辑、下架、取消上架、复制、删除。

### 7.5 套餐状态规则

套餐状态决定当前记录可执行的操作：

- 待上架 `pending`：表示已配置未来上架时间但尚未生效。该状态支持编辑，支持取消上架，支持删除；不支持复制，也不存在重新上架动作。
- 已上架 `active`：表示当前可售版本。该状态不支持编辑，支持下架，支持删除；不支持复制，也不存在重新上架动作。
- 已下架 `inactive`：表示套餐已被人工下架或待上架套餐被取消上架。该状态不支持编辑，不支持下架，不支持重新上架；支持复制，支持删除。
- 已被替换 `replaced`：表示套餐已被新版本替换，属于历史版本。该状态不支持编辑，不支持上下架，不支持重新上架；支持复制，支持删除。

状态计算规则：

- 若存储状态为 `inactive`，展示已下架。
- 若存储状态为 `replaced`，展示已被替换。
- 若存储状态为 `pending`，展示待上架。
- 若存储状态为其他值，则根据上架时间判断：上架时间晚于当前时间展示待上架，否则展示已上架。

### 7.6 新增套餐

入口：页面右上角「新增套餐」。

字段规则：

新增套餐弹窗包含以下字段：

- 套餐名称：必填，文本输入。
- 套餐类型：必填，可选择个人套餐、企业团购、加油包。
- 替换对象：条件必填。当上架时间为未来时必须选择；下拉选项仅展示同类型的已上架套餐。
- 关联 SKU：非必填，下拉仅展示中文 SKU 名称，提交时保存对应 `skuCode`。后续接入业务支撑系统后建议改为必填。
- 上架时间：必填，支持选择日期和时间。未来时间生成待上架套餐，当前或过去时间生成已上架套餐。
- 有效期：必填，可选择自然月、自然年、指定天数。选择指定天数时，天数必须为大于 0 的整数。
- 刊例价：个人套餐和加油包必填，必须为大于等于 0 的整数。
- 实际售价：个人套餐和加油包必填，必须为 0 到刊例价之间的整数。
- 起价：企业团购必填，必须为大于等于 0 的整数，单位为元/人/月。
- 起订数量：企业团购必填，必须为大于 0 的整数。
- 权益说明：非必填，支持添加多条文本。
- 适用渠道：只读字段，当前固定为自有渠道。
- 当前状态：只读字段，根据上架时间实时预览。

提交规则：

- 上架时间为未来：套餐状态为待上架，必须选择替换对象。
- 上架时间为当前或过去：套餐状态为已上架，提交后不可编辑。
- 替换对象必须与当前套餐类型一致。
- SKU 如与其他非历史套餐冲突，则提交失败；替换旧版本时允许复用被替换套餐的 SKU。
- 提交成功后关闭弹窗并刷新列表。
- 提交失败时展示错误提示，弹窗不关闭。

### 7.7 编辑套餐

入口：仅待上架套餐展示「编辑」。

规则：

- 仅待上架套餐允许编辑。
- 已上架、已下架、已被替换套餐不展示编辑入口。
- API 层需做同样校验，绕过前端提交时返回错误。
- 编辑时套餐类型不可修改。
- 编辑后如果上架时间仍为未来，保持待上架；如果上架时间调整为当前或过去，则可立即进入已上架，并按替换规则处理。

### 7.8 下架与取消上架

入口：表格操作列。

规则：

- 已上架套餐展示「下架」。
- 待上架套餐展示「取消上架」。
- 已下架套餐不允许重新上架。
- 已被替换套餐不允许上下架。

结果：

- 已上架套餐点击下架后，状态变为已下架。
- 待上架套餐点击取消上架后，状态变为已下架，不触发替换。
- 已下架套餐如需重新发布，只能复制后创建新版本。

### 7.9 复制套餐

入口：已下架、已被替换套餐展示「复制」。

规则：

- 点击复制后打开新增套餐弹窗。
- 默认带入原套餐名称、类型、SKU、有效期、价格、起订数量、权益说明等配置。
- 名称默认追加「副本」。
- 上架时间默认设置为当前时间 + 1 天。
- 复制提交后生成一条新套餐记录，不修改原套餐。
- 复制后的套餐如需待上架，仍需选择同类型替换对象。

补充说明：

- 当前项目代码仅对已下架、已被替换套餐展示复制入口。
- 若后续希望复制已上架套餐用于创建下一版，可扩展为 active 也展示复制，并默认将原套餐作为替换对象。

### 7.10 版本发布与生命周期同步

系统在查询套餐列表时触发生命周期同步：

1. 遍历套餐数据，补齐 `seriesKey`。
2. 找到状态为 `pending` 且上架时间已到达的套餐。
3. 将该套餐状态改为 `active`。
4. 将其 `replacesPackageId` 指向的旧套餐状态改为 `replaced`。
5. 同时将同一 `seriesKey` 下其他 active 套餐改为 `replaced`，避免同一售卖位存在多个 active。
6. 同步结果写回 localStorage。

### 7.11 删除规则

当前 MVP 允许删除任意状态套餐，删除前通过二次确认弹窗提示「删除后不可恢复」。

后续接入真实订单和权益履约后，建议调整为：

- 已上架套餐不允许物理删除。
- 已被替换套餐不允许物理删除。
- 已产生订单或权益实例的套餐只能逻辑删除或归档。

### 7.12 默认数据规则

当前 MVP 默认数据如下：

- 个人套餐：默认包含个人体验版、个人基础版、个人标准版、个人旗舰版。其中个人体验版实际售价为 0。
- 企业团购：默认仅保留企业团购标准版 1 条，起价为 49 元/人/月，起订数量为 10 席。
- 加油包：默认仅保留积分加油包 1K 1 条，有效期为 30 天。

mock 数据规则：

- `package_data_version` 当前为 `19`。
- 版本不一致或强制初始化时，重置 `package_templates` 为默认数据。
- 当前版本不保留旧 mock 数据迁移逻辑。

## 8. 验收标准

| 验收项 | 验收标准 |
| --- | --- |
| 筛选查询 | 输入关键词、选择类型或状态后，列表展示符合条件的数据 |
| 待上架校验 | 未来上架时间未选择替换对象时，提交失败 |
| 替换对象过滤 | 个人套餐只能选择个人套餐作为替换对象，企业团购和加油包同理 |
| 自动替换 | 待上架套餐到点后变为已上架，被替换套餐变为已被替换 |
| 编辑限制 | 只有待上架套餐显示编辑入口，API 也拒绝编辑非待上架套餐 |
| 下架限制 | 已下架套餐不能重新上架，只能复制创建新版本 |
| 默认数据 | 初始化后企业团购 1 条，加油包 1 条，加油包有效期 30 天 |
